## Architecture Notes

### Auth Flow

1. User submits credentials on `/login` (SvelteKit server action).
2. Server action calls `POST /auth/login` → API issues `athena_session` JWT cookie.
3. SvelteKit server action parses `Set-Cookie`, calls `cookies.set()` to forward.
4. Every dashboard load function calls `GET /auth/me` via `apiFetch` (forwards `athena_session` cookie).
5. 401 from API → `apiFetch` throws `redirect(303, '/login')`.

### Environment Variables

| Variable              | Used By                           | Value                                                        |
| --------------------- | --------------------------------- | ------------------------------------------------------------ |
| `API_BASE_URL`        | SvelteKit server-side             | `http://api:3000` (Docker) / `http://localhost:3000` (local) |
| `PUBLIC_API_BASE_URL` | Browser client (unused currently) | `http://localhost:3000`                                      |
| `JWT_SECRET`          | API                               | Set in `.env`                                                |
| `JWT_EXPIRY`          | API                               | Default `15m`                                                |

### Design System — "The Analytical Sanctuary"

- **Fonts:** Manrope (headlines), Inter (body).
- **Colours:** MD3 tokens — primary `#0056d2`, surface `#f9f9ff`, on-surface `#143161`.
- **Shadows:** `ambient-shadow` utility class (no 1px borders).
- **No external charting libraries** — all charts are pure CSS/SVG.

## Bugs Fixed

### 1. `GET /reporting/summary` → 500 (`column f.form_name does not exist`)

- **File:** `apps/api/src/routes/reporting.ts`
- **Root cause:** `public.forms` has a column called `display_name`, not `form_name`. The summary query and the `entities/:id/timeline` route both used `f.form_name`.
- **Fix:** Replaced `f.form_name AS source_form_name` → `f.display_name AS source_form_name` everywhere in the file.

### 2. `GET /indicators` and `GET /indicators/:id` → 500 (same root cause)

- **File:** `apps/api/src/routes/indicators.ts`
- **Fix:** Replaced both `f.form_name` references with `f.display_name`; updated `GROUP BY` clause accordingly.

### 3. Users and Devices dashboard pages always redirect back to `/dashboard`

- **Files:** `apps/web/src/routes/dashboard/users/+page.server.ts`, `apps/web/src/routes/dashboard/devices/+page.server.ts`
- **Root cause:** Both page loads checked `locals.user?.role` to guard admin-only access. In SvelteKit, child page server loads cannot reliably see mutations to `locals` made by a parent layout's server load, especially during client-side navigation where the parent may not re-run. The check always evaluated falsy, triggering the redirect.
- **Fix:** Replaced `locals.user` with `const { user } = await parent()` in both files. The layout's `load` already returns `{ user }`, making it correctly available to child loads via `parent()`.

## New Features

### API: `GET /reporting/activity`

Returns a rolling 7-day daily submission count for the authenticated org.

- Iterates all forms registered to the org via `getOrgForms(orgId)` helper.
- Uses `assertSafeIdentifier` on schema + form_key before constructing dynamic table names.
- Groups by `DATE_TRUNC('day', server_received_at)`, merges in JS.
- Always returns exactly 7 entries (zeros filled for empty days), ordered oldest → newest.
- Response shape: `Array<{ date: string; day: string; count: number }>`.

### API: `GET /reporting/recent`

Returns the 10 most recent submissions across all org form tables.

- Same iteration pattern as `/activity`.
- Merges all rows, sorts descending by `server_received_at`, takes top 10.
- Batch-resolves entity names (`metadata->>'name'`) and enumerator `display_name`/`email` using `WHERE id = ANY($1)` to avoid N+1 queries.
- Response shape: `RecentSubmission[]`.

### Types (`apps/web/src/lib/types.ts`)

Two new interfaces appended:

## Architecture Notes

- **`public.forms` column is `display_name`**, not `form_name`. Every SQL query joining forms must use `f.display_name`.
- **Submission tables are per-form, per-sector** (e.g. `wash_sector.submissions_water_point_baseline`). There is no unified submissions table. Org isolation is via `form_id` FK — submission tables have no `org_id` column.
- **Pattern for cross-table aggregation:** `getOrgForms(orgId)` → `assertSafeIdentifier` each identifier → query each table → merge and sort in JS. This pattern is used by `entities/:id/timeline`, `GET /reporting/activity`, and `GET /reporting/recent`.
- **SvelteKit child page loads must use `await parent()`** (not `locals`) to access data set by the parent layout's load function.
